# Human Playtest Review Template v1

## Purpose

This template captures human playtest results for AI_IF cases. It distinguishes case design issues from Game Master runtime issues and engine prompt improvements.

## Required Sections

### 1. Case Metadata

- Case ID:
- Title:
- Genre:
- Difficulty:
- Mode:
- Date played:
- Result:
- Was the case solved?

### 2. Case Outcome

Summarize:

- culprit named;
- victim/target;
- method;
- motive;
- opportunity;
- physical proof;
- final proof chain used by the player.

### 3. What Worked Well

Assess:

- contained setting;
- clue clarity;
- motive clarity;
- red herrings;
- emotional engagement;
- pacing;
- final reveal;
- player agency.

### 4. Main Gameplay Problems

Separate:

- case design problems;
- GM runtime behavior problems;
- engine prompt problems;
- image fidelity problems;
- voice input problems;
- player confusion;
- clue fairness problems.

### 5. GM Behavior Review

Check whether the GM:

- steered too much;
- summarized neutrally;
- used hidden hints;
- converted facts into theories;
- revealed interpretation too early;
- withheld observable evidence unfairly;
- handled early accusations correctly;
- distinguished plausible theory from proven accusation.

### 6. Evidence Fairness Review

For each major physical clue:

- Was the clue observable when fairly examined?
- Was interpretation delayed rather than observation?
- Did new details appear later without access/tool/lighting/permission change?
- Was any evidence unfairly hidden?

### 7. Case Board Review

Check whether case-board updates used:

- Known fact;
- Known claim;
- Known evidence;
- Unresolved significance.

Flag leading patterns:

- "Was something added?"
- "Who tampered?"
- "This points to..."
- "This suggests..."
- "Did the killer...?"

### 8. Image Review

Check:

- Did images preserve canonically present people?
- Did images omit important people or objects?
- Did images add non-canonical details?
- Did images reveal hidden clues?
- Were images correctly treated as visualization, not evidence?

### 9. Voice / Ambiguity Review

Check:

- Were misheard names clarified?
- Did the GM guess incorrectly?
- Were NPC names distinct enough?
- Did voice mode cause wrong turns?

### 10. Red Herring Review

For each major suspect:

- suspicious fact;
- innocent explanation;
- clearance clue;
- whether suspicion felt fair;
- whether unresolved suspicion remained after solution.

### 11. Final Proof Threshold Review

Confirm the solution required:

- motive;
- timing;
- opportunity/access;
- physical proof;
- contradiction or culprit mistake;
- weakening or clearing major alternatives.

### 12. Player Subjective Feedback

Record:

- Was it fun?
- Was it dull?
- Was it confusing?
- Did the player feel clever or steered?
- Was the reveal satisfying?
- Would the player want another case like this?

### 13. Recommended Case Revisions

List changes to the specific case only.

### 14. Recommended Engine Revisions

List changes to prompts, docs, schemas, runtime rules, image rules, or workflow.

### 15. Overall Assessment

Choose one:

- pass;
- pass with minor revision;
- pass with major revision;
- fail/rebuild;
- useful test but not replay-worthy.

Include a short explanation.
